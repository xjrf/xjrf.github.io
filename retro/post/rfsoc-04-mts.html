RFSoC学习 05 - 多Tile同步（MTS）
2006-02-16|RFSoC|RFSoCFPGAXilinx|⏱ 10 min | 3036 字

概述

在RFSoC中，每个Tile都有自己的时钟系统。这样做的好处是你可以让不同的Tile工作在完全不同的采样率下。但这也带来一个问题：当你需要让多个Tile协同工作时（MIMO或相控阵相关），它们之间的延迟难以对齐。

为什么需要MTS

在任何具有多个数模转换器和时钟结构的系统中，存在几个不确定的延迟来源：

  • 时钟分频器的初始相位
  • NCO的相位
  • FIFO的延迟
  • 时钟传输延迟
  • 数据传输延迟

RFSoC中的同步

传统的多芯片方案依赖JESD204B标准里的SYSREF信号来做同步。RFSoC虽然是单芯片方案，但为了兼容现有生态，也沿用了这套机制。只不过因为ADC/DAC和FPGA都集成在芯片内部，同步实现起来比多芯片方案简单。

虽然芯片内部集成消除了复杂的串行收发器链路，但每个转换器还是带了独立的FIFO。这些FIFO的作用是桥接PL时钟域和采样时钟域（会引入不确定的延迟，必须用SYSREF来测量和校准）。

MTS系统架构


外部时钟源 → DAC0_CLK_p/n → Tile 0 PLL → DAC0/1/2/3
          → DAC1_CLK_p/n → Tile 1 PLL → DAC0/1/2/3

外部SYSREF → SYSREF_p/n → 内部分配 → 所有Tile

PL时钟 → s0_axis_aclk (Tile 0的AXI-Stream接口时钟)
       → s1_axis_aclk (Tile 1的AXI-Stream接口时钟)

PL SYSREF → user_sysref_dac → IP同步状态机

关键信号

模拟域：

  • SYSREF_p/n：整个芯片只有一个，进来后内部分发给所有Tile
  • DACx_CLK_p/n / ADCx_CLK_p/n：每个Tile独立的时钟输入

数字域：

  • PL_clock：PL设计和tile的公共时钟
  • user_sysref_adc：PL给ADC Tile的SYSREF信号（同步到Tile0的PL时钟）
  • user_sysref_dac：PL给DAC Tile的SYSREF信号（同步到Tile0的PL时钟）

内部机制

  • 每个tile有一个内部模拟SYSREF延迟调整，通过允许在片上优化SYSREF建立/保持时间
  • 同步的逻辑由两部分配合完成：IP核自带的同步状态机，还有RFdc驱动API

MTS要求

RF-DAC MTS要求

  • Tile 0必须工作（主Tile，所有其他Tile以它为基准）
  • Tile 0的通道0必须启用
  • 参与MTS的所有tile必须：
    • 采样率相同
    • 抽取率相同
    • 每个AXIS周期的采样数相同
    • AXIS接口的时钟速率相同

RF-ADC MTS要求

  • Tile 0必须工作
  • Tile 0的通道0必须启用
  • 参与MTS的所有tile必须：
    • 采样率相同
    • 抽取率相同
    • 每个AXIS周期的采样数相同
    • AXIS接口的时钟速率相同
    • 属于相同类型（Quad或Dual）

禁用tile的处理

如果某个Tile在SYSREF的传播路径上但不需要使用，也需要做一些处理。根据不同器件，需要把它下电到特定状态：

  • Gen 1/Gen 2器件：下电到状态3
  • Gen 3/DFE器件：下电到状态6

这样SYSREF才能正常穿过这个Tile传到后面去。

MTS的同步步骤

整个同步过程大部分由硬件和驱动自动完成。

1. 使能所有时钟和SYSREF生成器

在同步开始之前，模拟和数字时钟都必须运行并锁定。（任何时钟更改都需要重新同步）

2. SYSREF模拟捕获

这一步是要让每个Tile都能可靠地捕获到SYSREF。芯片会自动调节内部的SYSREF延迟线，找到最优的建立/保持时间窗口。因为这个过程需要多次尝试，所以SYSREF必须是周期信号，不能是单脉冲。

3. 复位时钟分频器

等所有Tile都稳定后，下一个SYSREF边沿就会用来同步所有时钟分频器的相位。

4. FIFO延迟测量和调整

这一步用模拟SYSREF和PL_SYSREF配合，测量数据穿过FIFO花了多少时间。然后根据测量结果调整，让所有Tile的FIFO延迟一致。

5. 数字模块的同步更新

有些数字模块（比如NCO、QMC等）需要在特定时刻统一更新状态。这些更新也要用SYSREF来触发，保证所有Tile同时生效。

步骤2、3和4是自动的。

什么时候需要重新同步

以下情况需要重新启动同步过程：

  1. 使用API修改影响FIFO时钟速率或位宽的参数
  2. 参与同步的某个Tile被下电或重启
  3. 时钟发生任何更改

确定性延迟

上面说的MTS能保证多Tile之间对齐，但是，每次上电，整个通路的总延迟可能会有微小变化，变化量约等于FIFO读时钟周期除以每个周期读出的样本数。

如果应用要求每次上电延迟完全一致，需要用Target_Latency这个设置。

Target_Latency工作原理

  • 如果测出来的延迟小于目标值，会把它调整到目标值
  • 如果测出来大于目标值，会报错
  • MTS会忽略Target_Latency来保证所有tile对齐

确定Target_Latency值

  1. 先运行XRFdc_MultiConverter_Sync，目标设置为-1（只测量，不强制）
  2. 观察测出来是多少，再加上一点余量，作为以后的目标值

余量值：

  • RF-ADC tile：必须是FIFO读样本数 × 抽取因子的整数倍
  • RF-DAC tile：通常加16就行

示例


XRFdc_MultiConverter_Init(&DAC_Sync_Config, 0, 0, 0); // 初始化DAC
DAC_Sync_Config.Target_Latency = -1; // 首次运行，测量延迟
XRFdc_MultiConverter_Sync(&RFdcInst, XRFDC_DAC_TILE, &DAC_Sync_Config);

// 获取测量的延迟值
int measured_latency = DAC_Sync_Config.Latency[0];

// 设置目标延迟（测量值 + 余量）
DAC_Sync_Config.Target_Latency = measured_latency + 16; // RF-DAC余量=16

// 后续运行使用目标延迟
XRFdc_MultiConverter_Sync(&RFdcInst, XRFDC_DAC_TILE, &DAC_Sync_Config);

MTS接口端口

模拟SYSREF输入

端口名方向时钟域说明
sysref_in_p/n输入N/ASYSREF差分输入

PL SYSREF输入

端口名方向时钟域说明
user_sysref_adc输入m0_axis_aclkPL发给ADC的SYSREF，必须和ADC Tile0的PL时钟同步
user_sysref_dac输入s0_axis_aclkPL发给DAC的SYSREF，必须和DAC Tile0的PL时钟同步

应用场景

Massive MIMO

  • 64/128/256天线阵列
  • 每个天线需要独立的收发通道
  • 所有通道必须精确相位对齐用于波束成形

相控阵雷达

  • 多个发射/接收通道
  • 亚皮秒级相位对齐要求
  • 波束指向精度依赖于通道同步

分布式系统

  • 多个RFSoC器件协同工作
  • 跨器件的通道对齐
  • 确定性延迟要求

载波聚合

  • 多个频段同时发射/接收
  • 频段间相位关系
  • 时间对齐要求

调试和验证

验证MTS是否成功

  1. 检查API返回值
  2. 验证所有tile的延迟测量值
  3. 使用示波器或频谱仪验证输出信号相位
  4. 检查SYSREF捕获状态

常见问题

SYSREF捕获失败：

  • 检查SYSREF信号质量
  • 调整SYSREF延迟
  • 验证时钟已锁定

延迟不匹配：

  • 检查tile配置是否一致
  • 验证采样率、抽取率设置
  • 检查FIFO配置

重新同步失败：

  • 确保所有时钟稳定
  • 检查tile电源状态
  • 验证SYSREF周期性
